# Seedpeak Workflow

## 触发条件

当用户输入匹配以下模式时激活:
- `种子：<模糊灵感>` / `seed: <idea>`
- 用户表达一个模糊想法并希望结构化推进

不要因为用户没有完整描述就等待——从种子开始，一步步引导。

---

## 流程概览

```
1. 明确目标
   ├─ 种子 → 维度清单 → 逐维赋值 → 🚧 等待用户确认目标
2. 制定方案
   ├─ 任务拆分 → 执行次序 → 技术方案 → 🚧 等待用户确认方案
3. 补充上下文
   ├─ 角色/背景/输出要求/参考资料/禁忌 → 🚧 等待用户确认
4. 执行  ←─ 子任务可递归应用此流程
5. 整合
6. 反馈
7. 迭代  → 回到步骤 1，形成闭环
```

**🚧 门禁规则（重要）：每步之间必须等待用户显式确认**

每次你询问用户"这个方案/目标可以吗？"之后：
- ❌ 用户说"这里改一下"、"第二个任务不太对"、"再加一个步骤"——**这不是确认**，继续在本步骤迭代修改
- ✅ 用户说"可以"、"通过"、"没问题"、"就这样"、"go ahead"——**这才是确认**，可以进入下一步
- ✅ 用户主动说"下一步"、"继续"——**这也是确认**

用户任何讨论、修改、补充意见都不是确认信号。确认信号必须是明确的同意。

不要帮用户做决定。如果不是显式确认，就停在当前步骤不要前进。

---

## 流程说明

### 1. 明确目标

**目标：** 从一个模糊灵感（种子）逐步收敛为一个清晰、可执行的目标描述。

**你的任务：**

1. 收到种子后，先确认理解："收到，种子是「...」。我来帮你梳理成可执行的目标。"

2. 根据种子内容生成一个维度清单。维度因任务类型而异，典型维度包括:

| 维度 | 核心问题 |
| --- | --- |
| 领域/范围 | 在什么领域？边界在哪里？ |
| 具体目标 | 想达成什么具体成果？ |
| 可用时间 | 有多少时间？截止日期？ |
| 资源/预算 | 有什么资源可用？ |
| 当前起点 | 已经有什么基础？ |
| 约束条件 | 有什么限制？不能做什么？ |

3. **逐**个维度询问用户赋值，一次只问 1-2 个，不要一次性抛出全部。

4. 收集完毕后，汇总为最终目标描述。格式参考:

```markdown
## 最终目标

**一句话概括:** ...

**具体规格:**
- 领域/范围：...
- 成果：...
- 时间：...
- 约束：...
- ...
```

**🚧 门禁：** 必须等待用户**显式确认**目标准确后才能进入步骤 2。用户提出修改意见时，修改后重新呈现，继续等待确认。

---

### 2. 制定方案

**目标：** 将目标拆解为可执行的计划。

1. **任务拆分** — 将主任务拆为子任务清单
2. **执行次序** — 标注依赖关系，按优先级排序
3. **技术方案** — 明确使用的工具、技术、方法
4. **审核方案** — 向用户展示完整方案，询问是否可行

**你的任务：**

收到或确认目标后，主动生成方案草稿并呈现给用户。格式参考:

```markdown
## 执行方案

### 子任务（按执行顺序）
1. [任务 A] — 前置任务：无
2. [任务 B] — 前置任务：任务 A
3. [任务 C] — 前置任务：任务 A
...
> 每个任务用一句话描述，标注依赖关系。颗粒度建议：每个任务可在 1-2 小时内完成。

### 工具与技术
- 工具名称：用途说明
- 是否需要本地安装：是/否
- 是否需要外部账号/权限：是/否
> 列出完成各任务所需的主要工具、框架、API 等。

### 可行性与风险
- 风险点：... → 应对措施：...
- 不确定因素：... → 验证方式：...
> 主动暴露可能的坑，不要执行到一半才说"这个做不了"。

### 交付物与验收标准
- 交付物：[具体产出，如文件、代码、文档]
- 验收标准：[可衡量的成功标准，如测试通过、格式符合要求]

**这个方案可以吗？需要调整吗？**
```

**🚧 门禁：** 呈现方案后，必须等待用户**显式确认**才能进入下一步。用户说"这里改一下"、"第二个任务不对"等修改意见时，修改后重新呈现，继续等待确认。只有用户说"可以"、"通过"、"没问题"、"就这样"才算确认通过。

如果用户拒绝方案，先问清楚哪里不满意，针对性修改后重新提交审核。

---

### 3. 补充上下文

**目标：** 补齐执行所需的所有背景信息，提高输出质量。

**你的任务：**

逐一确认以下上下文维度。同样一次问 1-2 个，不要让用户有压力:

| 类别 | 核心问题 | 示例 |
| --- | --- | --- |
| 角色设定 | AI 以什么身份产出？ | "你是一位资深产品经理" |
| 背景信息 | 前因后果？已有材料？ | "公司预算 10 万，第三版修改" |
| 输出要求 | 格式、长度、风格？ | "输出 JSON，500 字以内" |
| 参考资料 | 可以参考什么？ | "参考附件中的销售数据" |
| 禁忌边界 | 绝对不能做什么？ | "不要出现具体品牌名称" |

用户可能已经在之前步骤中提供过部分上下文，直接复用，不要重复询问。

---

### 4. 执行

**目标：** 按方案执行每个子任务。

**重要规则：**
- 执行前先回顾方案和上下文，确保没有遗漏
- 如果子任务足够复杂，**递归应用此 7 步流程**（即对子任务单独执行步骤 1-7）
- 执行过程中主动反馈进度:
  - ✅ 任务 A 完成
  - ⏳ 任务 B 进行中
  - ❌ 遇到问题：...
- 遇到困难时主动暂停并询问用户意见

---

### 5. 整合

**目标：** 将所有子任务产出整合为统一的最终产物。

- 检查各部分之间是否衔接一致
- 消除重复、填补缺口
- 确保整体结构和格式符合第 3 步确定的输出要求
- 呈现整合后的完整产物给用户

---

### 6. 反馈

**目标：** 对自己产出的质量进行自检，并邀请用户评价。

**你的任务：**

1. 先做自检:
   - 是否满足第 1 步确定的所有目标？
   - 是否符合第 3 步确定的输出要求？
   - 是否有明显的缺陷或遗漏？

2. 然后邀请用户反馈:
   > "请检查这份产出。有什么需要修改或补充的地方吗？"

---

### 7. 迭代

**目标：** 根据反馈持续改进，直到满意。

- 修复用户指出的缺陷
- 根据补充内容更新产出
- 如有必要，回到步骤 1（明确迭代目标），重新走流程
- 每次都形成闭环：目标 → 执行 → 反馈 → 再目标

---

## 常见错误与注意事项

| 错误做法 | 正确做法 |
| --- | --- |
| 一次性抛出所有维度让用户填写 | 一次只问 1-2 个维度，逐步收集 |
| 用户给种子后直接开始执行 | 必须先走完目标→方案→上下文再执行 |
| **用户讨论方案时自动进入下一步** | **识别讨论≠确认，停在当前步骤等待显式确认** |
| 方案被拒绝后直接重做 | 先问用户哪里不满意，针对性修改 |
| 完整执行完才给用户看结果 | 每个步骤完成时主动同步 |
| 忽略用户已在前面说过的信息 | 复用已有信息，不重复询问 |
| 子任务复杂时仍然线性执行 | 对复杂子任务递归应用此 7 步流程 |

如果用户偏离流程（例如跳过步骤直接要求执行），温和提醒并建议先完成前置步骤，但尊重用户的选择。
